# Testing Wisdom

<!-- Universal testing patterns across any project -->

- Source verification tests (readFileSync + regex) catch structural regressions that runtime tests miss. [from: BoxBox]
- Dual-layer testing (source verification + behavioral) catches more bugs than either alone. Source-only can all pass while implementation has bugs. [from: BoxBox]
- Every new test file needs framework-specific mock setup. Document the required mocks in AGENTS.md. [from: BoxBox]
- For features with 3+ table joins, write a "data chain smoke test" that verifies the full FK path. [from: BoxBox]
- Tests prove logic works. They don't prove the product works. Always verify with real data after building. [from: BoxBox]
- Parameterized `it.each` source tests work well for sweeping refactors across many files. [from: BoxBox]
- Assert error code AND message: `.rejects.toMatchObject({ code: 'FORBIDDEN' })`, not `.rejects.toThrow()`. Weak assertions catch wrong errors. [from: BoxBox]
- Extract pure functions from DB-heavy code for unit testability without mocks (e.g., `calculatePercentageChange`, `isValidStatusTransition`). [from: BoxBox]
- When adding ownership/auth checks to existing routes, update ALL existing tests — they need the new mock expectations. [from: BoxBox]
- When removing fields from input schema, update ALL test files constructing input objects for that procedure. [from: BoxBox]
- Serialize-and-assert-absent for sensitive fields: JSON-stringify the DTO and assert zero substring matches against a FORBIDDEN list (answer keys, explanations). [from: itqan]
- Run the forbidden-field leak assertion against EVERY variant/shape of a DTO, not just the canonical one. [from: itqan]
- The iteration journal (PROGRESS.md) is the TDD oracle for RED-before-GREEN ordering — not the final diff. [from: itqan]
- Add an inertness regression test for any calc/eval engine: assert it has >=1 live caller. [from: itqan]
- Prefer property tests (50+ runs, deterministic low-discrepancy RNG) over fixed-value tests for samplers/generators; assert a statistical bound (e.g. collision rate < 5%). [from: itqan]
- Live e2e against a seeded environment catches races and cache-coherence bugs that mocked tests structurally cannot. [from: itqan]
- Negative-regex source tests (e.g. `expect(src).not.toMatch(/animate-/)`) are a first-class design-system gate, not a brittle nitpick — they assert a constraint is NEVER violated and catch accidental reintroductions. Keep them. [from: itqan]
- CSS class-string assertions in source tests encode the design-handoff pixel contract (color/spacing/type) and catch visual regressions without a browser; assert class membership, not full attribute strings, to avoid trailing-space brittleness. [from: itqan]
- Build the sanitized client DTO from an explicit ALLOW-LIST of fields (whitelist), never by deleting a denylist of forbidden keys. A new sensitive field added to the model is then excluded by default, instead of leaking until someone remembers to add it to the denylist. Test the whitelist construction, not just the absence of known-bad keys. [from: itqan]
- Test the leak property at BOTH layers — the privileged preview path AND the user-facing sanitized path. The gold-standard assertion proves the server reads by a TRUSTED server-side key (e.g. the DB `imageKey`), never the client-supplied id: `expect(storage.get).toHaveBeenCalledWith(dbKey)` AND `expect(storage.get).not.toHaveBeenCalledWith(clientId)`, with the sensitive field asserted `undefined` in the returned DTO. [from: itqan]
- Assert pure functions do NOT mutate their inputs: snapshot/clone the argument before the call, run it, then assert the original array/object is unchanged. A sampler/selection/scoring engine that `splice`s or sorts its input in place corrupts the caller silently and passes every output-only test. [from: itqan]
- A test that asserts the CURRENT (buggy) output as 'expected' locks the bug in: a passing test is not proof of correctness if the assertion was written to match observed behavior rather than the spec. A function fed degenerate inputs (empty attempts/counts) can return a plausible-looking constant; assert against the SPEC value with realistic inputs, not against whatever the code emits today. [from: itqan]
- Mocks structurally cannot catch concurrency/atomicity bugs; audit reward/balance/status write-paths for atomicity FIRST, and prove the race with a live-DB (or distinct-transaction-client) RED test before fixing. [from: ITQAN]
- Pin the clock (setSystemTime/fake timers) for any time- or timezone-dependent test; evening/midnight/tz flakiness is the smell that an ambient clock leaked into the unit under test. [from: ITQAN]
- Dual-path leak test gold standard: write a CONTROL test proving the matcher detects a deliberately-leaky payload FIRST, then run the real assertion; scan field NAMES and rendered secret VALUES, not just key presence. [from: ITQAN]
