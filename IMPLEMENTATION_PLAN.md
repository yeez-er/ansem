**Key architectural decisions** (from specs — do not relitigate):

- Snapshots are the source of truth; boards are derived. Daily score = `computeScore(windowDelta(baseline, latest))`, never summed rows — duplicate snapshots cannot double-count (specs 001/006).
- Counts are `bigint` end-to-end; serialized as strings at the API boundary (X views overflow int4 / 2^53).
- Upserts key on natural keys (`UNIQUE(platform, handle)`, `UNIQUE(platform, platform_post_id)`) — never on uuid PKs, or `onConflictDoNothing` never triggers.
- Pure logic (URL parsing, scoring) lives in `src/lib/` with no I/O and no ambient clock; providers live behind `MetricsProvider` and NEVER throw.
- Public DTOs are allow-list constructed; single-entity "no data" lookups return `null`, never `{}`. Lists return `[]` (a list is a list).
- **Build order constraint (spec 009)**: 000 → 009 Part A → 002 → … → 009 Part B. Seed (010) lands after the leaderboard query layer (007) and BEFORE any UI task (008 / 009B UI).
- TikTok short links are resolved server-side at submit time (spec 002, decided round 3); `needsResolution` never reaches persistence; for `needsResolution` parser results, `canonicalUrl` = the normalized short link `https://vm.tiktok.com/<code>` (pinned, spec-review note #1); resolution attempts count toward the rolling 24h rate limit and the quota check runs BEFORE the fetch (spec-review note #2).
- Third-party scraper risk accepted 2026-07-05 (Yasser): SocialData.tools (X refresh), Apify actors (TikTok/IG); adapters built against recorded fixtures in v1. ScrapeCreators is the swap-in fallback vendor.
- All date math UTC; board day boundary 00:00:00 UTC. Clock is pinned (fake timers) in every date-touching test.

### Task 6: `submissions.submit` procedure — dedupe, rate limit, short-link resolution

- **Spec**: 002 · **Complexity**: L · **Dependencies**: 3, 4, 5 · **Blocks**: 25, 26
- **Acceptance criteria**:
  - Unparseable URL → `BAD_REQUEST` / `UNSUPPORTED_URL`; unauthenticated → `UNAUTHORIZED`; banned creator → `FORBIDDEN` / `CREATOR_BANNED` (assert codes with `.toMatchObject`, never bare `.toThrow()`)
  - Duplicate submission returns `alreadyTracked: true`, creates NO new row (assert row count); does not consume quota
  - Two concurrent submissions of the same URL → exactly one post row (distinct-transaction test, not mocked); upsert + insert in ONE transaction
  - 21st insertion in rolling 24h → `TOO_MANY_REQUESTS`; resolution attempts also count toward the limit; quota check runs before the redirect fetch
  - Short-link dedupe: same video via `vm.tiktok.com/<code>` and canonical URL → ONE row (mocked redirect; `redirect: 'manual'`, `Location` header only, 5s timeout, try-catch); unresolvable (network failure / non-TikTok redirect) → `BAD_REQUEST` / `UNRESOLVABLE_URL`, no creator or post row created
  - Placeholder creator uses deterministic handle `placeholder:<platformPostId>`, `profile_url` = canonical post URL stand-in; `AUTO_APPROVE_SUBMISSIONS=true` inserts `status: 'approved'` (env var added to `.env.example` + `src/env.ts`)
- **Test strategy**: integration tests against real test DB (`@vitest-environment node`), distinct-transaction concurrency test, mocked `fetch` for resolution; fake timers for the 24h window.
- **Files to create**: `src/server/api/routers/submissions/submit.ts`, `submissions/index.ts`.
- **Files to modify**: `src/server/api/root.ts` (register), `src/env.ts`, `.env.example` (`AUTO_APPROVE_SUBMISSIONS`).
- **Progress (2026-07-05, scoped iteration 1 of 2)**: Core flow landed — `UNSUPPORTED_URL`/`UNAUTHORIZED`/`CREATOR_BANNED` typed errors (banned wins over `alreadyTracked`), transactional creator-upsert + post-insert on the natural-key `onConflictDoNothing` gates (concurrent duplicate → exactly one row, proven distinct-transaction), IG placeholder creators (`placeholder:<id>`, canonical-URL profile stand-in), short-link resolution (one hop, `redirect: 'manual'`, 5s timeout, try-catch; short↔canonical dedupe to one row; `UNRESOLVABLE_URL` on network failure/non-TikTok/no-Location), `AUTO_APPROVE_SUBMISSIONS` env flag (boot-validated as strict "true"/"false"). Infra: shared `src/tests/helpers/test-db.ts` harness (extracted on 2nd occurrence; schema-runtime refactored onto it) + `fileParallelism: false` so DB suites never race on `TEST_DATABASE_URL`.
- **Progress (2026-07-05, iteration 2 of 2 — DONE)**: Rolling 24h rate limit landed. Design question settled: new `resolution_attempts` table (schema + barrel + generated migration `0001`) records an attempt BEFORE the redirect fetch, so failed resolutions consume quota despite leaving no post row; a successful insert deletes its attempt in the same transaction (one submission = one quota unit; deduped short links keep the attempt — the fetch is never free). Quota = in-window inserted posts + in-window attempts, DB-derived per call, gated once right after parse (21st → `TOO_MANY_REQUESTS`, blocked short link costs zero outbound fetches, duplicates consume nothing, per-user isolation, window rolls at 24h). **Deviation from planned test strategy**: no fake timers — the procedure's own rows get timestamps from Postgres `defaultNow()`, which fake JS timers can't touch; tests seed backdated rows relative to real now (±1h slack), fully deterministic. Two YELLOW debts self-filed in `KNOWN_ISSUES.md`: quota gate is read-then-insert (burst overshoot possible), and `resolution_attempts` has no pruning.
